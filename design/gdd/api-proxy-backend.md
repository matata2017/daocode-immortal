# API代理后端 (API Proxy Backend)

> **Status**: Designed
> **Author**: User + Claude Code
> **Last Updated**: 2026-03-30
> **Implements Pillar**: 安全性、性能

## Overview

**API代理后端**是一个独立的后端服务，负责代理所有 MiniMax AI API 调用，保护 API Key 不暴露在客户端。

服务提供请求转发、Redis 缓存、用户限流、错误重试、日志记录等功能。客户端（Godot 游戏）通过内部 API 调用后端，后端再调用 MiniMax API。

**为什么需要这个系统**：MiniMax API Key 是敏感信息，绝对不能暴露在客户端代码中。没有代理后端，任何反编译都能获取 Key，造成安全风险和费用损失。

## Player Fantasy

**玩家无感知，但体验流畅。**

API 代理是"基础设施玩家不会直接与之交互，但它的性能直接影响 AI 面试官的响应速度。

### 玩家体验场景
**场景 1：快速响应**
玩家答完题，AI 面试官在 1 秒内开始回复。Redis 缓存命中时几乎是即时响应。

**场景 2：限流保护**
玩家疯狂点击"重新生成"按钮。系统显示"请求过于频繁，请稍后再试"。防止 API 滥用。

**场景 3：离线降级**
网络断开时，系统使用预设回复模板，面试继续进行。

### 支撑的游戏支柱
| 支柱 | 如何体现 |
|------|----------|
| **沉浸修仙** | 快速响应让对话自然流畅 |
| **实用至上** | AI 回复包含真实面试建议 |
| **安全性** | API Key 永不暴露 |

| 组件 | 技术选择 | 理由 |

### States and Transitions

#### 请求处理流程
```
┌─────────────────┐
│  请求到达       │
└────────┬────────┘
         │ 验证请求
         ↓
┌─────────────────┐     无效
│  请求验证       │ ─────────→ 400 Bad Request
└────────┬────────┘
         │ 有效
         ↓
┌─────────────────┐     超限
│  限流检查       │ ─────────→ 429 Too Many Requests
└────────┬────────┘
         │ 通过
         ↓
┌─────────────────┐     命中
│  Redis 缓存查询  │ ─────────→ 200 Cached Response
└────────┬────────┘
         │ 未命中
         ↓
┌─────────────────┐     失败
│  MiniMax API    │ ─────────→ 重试 (最多 3 次)
└────────┬────────┘
         │ 成功
         ↓
┌─────────────────┐
│  缓存结果       │
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│  200 Response   │
└─────────────────┘
```

## Formulas

### 缓存 Key 生成
```
cache_key = SHA256(boss_id + prompt_hash + context_hash)

# 示例
# cache_key = "a1b2c3d4e5f6..."
```

### 限流算法（令牌桶）
```
# 每用户每分钟 10 次请求
rate_limit(user_id) =
    current_tokens = redis.get("rate_limit:{user_id}")
    IF current_tokens <= 0:
        RETURN false  # 被限流
    ELSE:
        redis.decr("rate_limit:{user_id}")
        RETURN true  # 允许

# 令牌恢复：每 6 秒恢复 1 个令牌
# Lua 脚本实现原子操作
```

### 缓存命中率
```
cache_hit_rate = cached_requests / total_requests × 100%

# 目标：> 30% (热门问题缓存命中)
```

### 响应时间预算
```
total_latency =
    redis_latency (目标: < 10ms) +
    minimax_latency (目标: < 2000ms) +
    processing_overhead (目标: < 50ms)

# 总目标: < 2100ms (P95)
```

## Edge Cases
| 边缘情况 | 处理方式 | 影响范围 |
|----------|----------|----------|
| **MiniMax API 超时** | 返回预设回复模板，记录日志 | AI 集成 |
| **MiniMax API 错误** | 重试 3 次，失败则返回降级回复 | AI 集成 |
| **Redis 连接失败** | 跳过缓存，直接调用 API | 性能 |
| **请求体过大** | 拒绝请求，返回 413 | API 网关 |
| **无效 API Key** | 记录安全日志，返回 500 | 安全性 |
| **DDoS 攻击** | 全局限流 + IP 黑名单 | 安全性 |
| **并发请求过多** | 请求队列 + 超时拒绝 | 性能 |
| **缓存雪崩** | 随机 TTL 偏移 | 性能 |

## Dependencies
### 上游依赖（无）
API 代理后端是基础设施层，**无上游依赖**。

### 下游依赖（被依赖）
| 系统 | 依赖类型 | 接口契约 |
|------|----------|----------|

### 外部依赖
| 服务 | 依赖类型 | 说明 |
|------|----------|------|
| **MiniMax API** | 硬依赖 | AI 面试官对话生成 |
| **Redis** | 软依赖 | 缓存失效时仍可工作 |

## Tuning Knobs
| Knob | 类型 | 默认值 | 范围 | 影响范围 |
|------|------|--------|------|----------|
| `rate_limit_per_minute` | int | 10 | 5-100 | 每用户每分钟请求限制 |
| `cache_ttl_seconds` | int | 3600 | 300-86400 | 缓存过期时间 |
| `max_request_size_kb` | int | 64 | 16-256 | 最大请求体大小 |
| `minimax_timeout_ms` | int | 5000 | 3000-30000 | MiniMax API 超时 |
| `retry_count` | int | 3 | 1-5 | 失败重试次数 |
| `retry_delay_ms` | int | 500 | 100-2000 | 重试间隔 |
| `max_concurrent_requests` | int | 100 | 50-500 | 最大并发请求数 |

### 调参指南
- **限流值**：若发现用户频繁触发限流，可提高 `rate_limit_per_minute`
- **缓存时间**：若 AI 回复变化频繁，可降低 `cache_ttl_seconds`
- **超时时间**：若 MiniMax 响应慢，可提高 `minimax_timeout_ms`
- **并发数**：若服务器压力大，可降低 `max_concurrent_requests`
